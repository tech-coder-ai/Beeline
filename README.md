# Beeline

Beeline is an enterprise AI Data Intelligence Platform: ask business questions in
plain English and get governed, explainable Hive SQL, safe execution, and
adaptive visualizations back — not a chatbot that hallucinates queries.

The LLM never generates executable SQL directly. Every question flows through a
chain of independent, auditable stages:

```
User
 -> Query Refiner        (spelling, abbreviations, glossary resolution)
 -> Intent Engine         (classify what's being asked)
 -> Semantic Search        (catalog, glossary, metrics, SQL knowledge library)
 -> Clarification Engine   (asks instead of guessing when confidence is low)
 -> Query Planner          (LLM proposes tables/columns/filters — verified against the catalog)
 -> SQL Generator           (plan -> Hive SQL; deterministic fallback if the LLM is down)
 -> SQL Validator            (sqlglot-parsed: SELECT-only, no comments, no stacked statements)
 -> SQL Optimizer             (LIMIT injection, partition-filter advisories)
 -> Cost Estimator             (EXPLAIN + catalog stats; blocks runaway scans before execution)
 -> Executor                    (Hive, with result caching)
 -> Result Interpreter           (grounded narrative + insights)
 -> Visualization Planner         (deterministic chart selection — never LLM-chosen)
 -> BeelineResponse                (single structured schema; Angular decides how to render)
```

## Stack

| Layer | Tech |
|---|---|
| Frontend | Angular 20 (standalone + signals), Angular Material, AG Grid, Apache ECharts, SCSS glassmorphism |
| Backend | FastAPI, async SQLAlchemy, Pydantic, Alembic, sqlglot |
| Analytics source (v1) | Apache Hive via PyHive |
| Metadata repository | SQLite (dev) / PostgreSQL (recommended) — config only, no code changes |
| Cache | Redis (transparent in-memory fallback if unavailable) |
| LLM | Pluggable: OpenAI-compatible, or **Stellar** (custom endpoint: `POST {systemPrompt, userMessage}`) |

Deliberately excluded from v1 (see PRD decisions): Celery, RabbitMQ, Kafka,
LangChain/LangGraph, RBAC/SSO/JWT, Prometheus/OpenTelemetry, Kubernetes, CI/CD.
The architecture has clean extension points for all of these later.

## Repository layout

```
backend/
  app/
    connectors/        IAnalyticsConnector abstraction + Hive implementation
    llm/                Provider abstraction: OpenAI-compatible + Stellar
    pipeline/           Orchestrator + one module per pipeline stage
    services/           Metadata sync, enrichment, approvals, chat, query library
    api/v1/             REST routers (chat, metadata, workspace, admin)
    models/             SQLAlchemy models (catalog, semantic, chat, governance...)
    schemas/            Pydantic DTOs, including the BeelineResponse contract
  alembic/              Migrations (autogenerated from models)
  sample_data/          Generator + seed.hql for the demo Hive dataset
  tests/                pytest suite (guard rails, optimizer, SQL builder, viz planner, LLM providers)
frontend/
  src/app/
    core/               API client, chat state (signals), models, theme
    shared/             Adaptive response renderer, chart/grid/KPI components
    features/           chat, metadata, dashboards, queries, admin
docker-compose.yml       Postgres, Redis, standalone Hive + seeding job, backend, frontend
```

## Local development (no Docker)

Backend:

```bash
cd backend
uv venv --python 3.12 .venv        # or python3.12 -m venv .venv
uv pip install -e ".[dev]" --python .venv/bin/python
.venv/bin/uvicorn app.main:app --reload --port 8010
```

The backend defaults to a local SQLite metadata repository
(`backend/beeline_meta.db`, created automatically) and expects Hive at
`localhost:10000`. Edit `backend/config/settings.yaml` or set
`BEELINE__CONNECTORS__DEFINITIONS__HIVE__HOST=...` style env overrides.

Frontend:

```bash
cd frontend
npm install --legacy-peer-deps
npm run start -- --port 4210       # proxy.conf.json forwards /api to :8010
```

Open http://localhost:4210.

## Docker Compose (full stack incl. Hive)

```bash
cp .env.example .env      # add OPENAI_API_KEY or STELLAR_ENDPOINT
docker compose up --build
```

This starts Postgres (metadata repository), Redis, a standalone HiveServer2
(embedded Derby metastore — demo/dev only), a one-shot `hive-init` job that
loads ~4,300 rows of generated sample sales data (`sales.dim_customers`,
`sales.dim_products`, `sales.fact_sales`), the backend, and the frontend.

- Frontend: http://localhost:8080
- Backend API docs: http://localhost:8000/api/docs

After the stack is up, trigger a metadata sync from **Admin → Connectors & Sync
→ Incremental sync** (or `POST /api/v1/admin/sync`) so the catalog picks up the
seeded Hive tables — the NL pipeline only ever reads the synchronized catalog,
never the live Hive metastore.

## Configuring the LLM provider

`backend/config/settings.yaml` → `llm.active` selects the provider.

- `openai` — any OpenAI-compatible chat completions endpoint (OpenAI, Azure,
  vLLM, Ollama...). Set `OPENAI_API_KEY`.
- `stellar` — your local endpoint. Beeline POSTs
  `{"systemPrompt": "...", "userMessage": "..."}` and reads the reply from the
  JSON field named by `response_field` (dot-path supported, e.g. `data.output`).
  Set `STELLAR_ENDPOINT`.

If the active LLM is unreachable, the platform degrades gracefully: intent
classification falls back to keyword heuristics, SQL generation falls back to
a deterministic plan-to-SQL builder, and the failure is surfaced as a warning
in the UI rather than a hard error.

## Guard rails (always on, not configurable away)

Read-only, single-statement, SELECT-only SQL is enforced by real parsing
(`sqlglot`), not regex matching alone: blocked keywords, no comments, no
stacked statements, no cartesian joins, bounded join/subquery depth, and
(when a catalog is loaded) rejection of any table Beeline doesn't recognize.
See `backend/tests/test_guardrails.py`.

## Extending the platform

- **New analytics connector**: implement `IAnalyticsConnector` (+
  `ISQLDialect`, `IMetadataProvider`, `IStatisticsProvider`, `IQueryEstimator`)
  under `backend/app/connectors/<name>/`, register with
  `@register_connector("name")`, add a `connectors.definitions.<name>` block to
  `settings.yaml`. No core pipeline changes required.
- **New LLM provider**: implement `ILLMProvider.complete()`, call
  `register_provider("name", MyProvider)`, add a `llm.providers.<name>` config
  block.
- **New chart type**: extend `VisualizationPlanner` (backend, deterministic
  selection) and `ChartComponent` (frontend, ECharts option builder).
- **New grid action button**: add an entry to `api_actions` in
  `settings.yaml` — no code change.
